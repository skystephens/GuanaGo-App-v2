Sección 03

Embudo de Ofertas — De la tabla Promociones al turista

El embudo alimenta dos audiencias: turistas en la isla (necesitan actuar hoy) y turistas que planean viajar (necesitan inspiración y urgencia futura). Usa tablas que ya existen.

🏝 Audiencia 1 · Turista en la isla

Quién: Turista con app instalada, ya en San Andrés.
Necesidad: Oferta para HOY o esta semana.
Canal: Push FCM + in-app banner + WhatsApp si se registró.
Urgencia: Alta — vencimiento en horas o días.

✈️ Audiencia 2 · Turista planificando viaje

Quién: Lead en tabla Leads con Fechas_del_Viaje futuras.
Necesidad: Ofertas para cuando llegue + paquetes anticipados.
Canal: Email + WhatsApp.
Urgencia: Media — 30/60/90 días antes del viaje.

Flujo completo del embudo
01
Aliado crea la promoción
Desde su panel Partner, el aliado llena: título, descripción, tipo (descuento % / 2x1 / incluye extra), vencimiento, y qué servicio aplica. Se registra en Promociones con Estado = borrador.
Promociones tblBNPE6
02
Sky aprueba (o auto-aprueba con reglas)
Sky cambia Estado → activa en Airtable. Opcionalmente: auto-aprobar si el aliado tiene plan Premium y la promo no supera un % máximo. Make escucha el cambio de estado.
Make Watch trigger
03
Make identifica las dos audiencias
Make consulta Perfiles_Usuarios y Leads:
— Audiencia 1: TipoPerfil = Turista + Estado = activo (en la isla ahora)
— Audiencia 2: Leads.Fechas_del_Viaje en los próximos 90 días
Segmentación
04
Construye el mensaje personalizado por canal
Para push: corto, urgente, emoji. Para email: imagen del aliado + descripción + CTA "Reserva ahora". Para WhatsApp: texto directo + link al microsite del aliado.
Contenido dinámico
05
Deposita en Notificaciones_Queue
Cada mensaje va como un record en Notificaciones_Queue con Canal, Destinatario_ID, Mensaje y Estado = pendiente. El escenario de envío corre cada 15 min.
Notificaciones_Queue
06
Entrega multi-canal
FCM push → app. WhatsApp Cloud API → número de usuario. Email → SendGrid/Alegra. Cada envío actualiza Estado → enviado con timestamp.
Entrega
07
Turista click → microsite del aliado → reserva o visita
El link lleva a guanago.travel/[slug]. Si reserva: PayU → GP automáticos. Si solo visita: check-in en mapa → +15 GP. La promo registra uso_count.
+GP · caja registradora

Sección 04

Escenarios Make.com para el embudo

Todos usan tablas existentes. Ninguno toca flujos activos de pago o cotizaciones.

💰

Escenario: Promo aprobada → Segmentar audiencias → Notificar

Prioridad Alta
Watch Promociones: Estado → activa→ Leer datos aliado (Directorio_Mapa via AliadoID)→ Query Perfiles_Usuarios: TipoPerfil=Turista + activo→ Query Leads: Fechas_del_Viaje ≤ 90 días→ Router por canal (push / WhatsApp / email)→ Create records Notificaciones_Queue→ Update Promo: notificados_count++
📅

Escenario: Cron diario 8am → Promos del día → Flash push

Prioridad Alta
Cron 8:00 am diario→ Buscar Promociones activas que vencen HOY→ Para cada promo: construir mensaje urgente→ Depositar en Notificaciones_Queue (Canal=push)→ FCM broadcast a turistas activos en isla
✈️

Escenario: Cron semanal → Leads pre-viaje → Email de inspiración

Prioridad Media
Cron lunes 9:00 am→ Query Leads: Fechas_del_Viaje entre hoy + 7 y hoy + 90→ Query Promociones: Estado=activa + no vence en 48h→ Query Directorio_Mapa: aliados Premium destacados→ Armar email HTML "Tu viaje se acerca — estas ofertas te esperan"→ Depositar en Notificaciones_Queue (Canal=email)
🗺

Escenario: Nuevo aliado Familia B → Microsite + Directorio_Mapa → Pin en mapa

Prioridad Media
Nuevo record Directorio_Mapa: Es_Aliado=true→ Auto-generar ID_Slug (si vacío)→ Crear doc Firestore /aliados/{slug} (solo campos públicos)→ Si tiene Catalogo_Productos: sync a Firestore /aliados/{slug}/catalogo→ Si tiene Menu_Items: sync a Firestore /aliados/{slug}/menu→ Crear post bienvenida en Contenido_Redes (Estado=Aprobado)
⭐

Escenario: Check-in en mapa → GP inmediatos → Push confirmación

Prioridad Media
Firestore trigger: nuevo check-in usuario→ Identificar aliado del POI (Directorio_Mapa)→ Sumar GP en Perfiles_Usuarios.GuanaPoints→ Log en GUANA_Transacciones: Tipo=check_in, Monto=+15→ Push: "¡Ganaste 15 GP en [nombre aliado]! 🌴"→ Si promo activa del aliado: mostrar en notif
🔄

Escenario: Promo vencida → Desactivar + Log + Reporte aliado

Mantenimiento
Cron: cada hora→ Buscar Promociones: Vencimiento < ahora + Estado = activa→ Update Estado → vencida→ Actualizar Firestore: promo_activa = false→ Email aliado: "Tu promo terminó — X personas la vieron, Y hicieron click"

⚠️ Recordatorio Airtable MCP

create_table no existe en el MCP — las tablas Menu_Items y Catalogo_Productos se crean manualmente en la UI de Airtable. Luego se comparte el tblXXXX ID para conectar desde Make y el backend.

Campos Directorio_Mapa nuevos = safe

Todos los campos nuevos son opcionales (no tienen validación requerida). Los 31 registros existentes no se rompen — simplemente quedan vacíos en los campos nuevos hasta ser llenados.

⚠️ Campos privados — nunca exponer

Comision_Pactada, Saldo_Puntos_Empresa, Wallet_Hedera en Directorio_Mapa. El endpoint público del backend debe filtrarlos en cada query antes de enviar a Firestore.
